# Admin Guide

This guide is for administrators deploying dirblock on managed Linux systems, including headless multi-user hosts where users primarily connect over SSH.

## Deployment Model

A managed install usually has:

```text
/usr/bin/dirblock
/etc/dirblock/dirblock.toml
```

The system config can use `~` in watched paths:

```toml
[watched]
"~/.ssh" = [
    "/usr/bin/ssh",
    "/usr/bin/git",
]
```

When a regular user runs `/usr/bin/dirblock`, `~` expands to that user's home directory. This lets one managed `/etc/dirblock/dirblock.toml` act as a per-user policy template:

```text
alice runs dirblock -> ~/.ssh means /home/alice/.ssh
bob runs dirblock   -> ~/.ssh means /home/bob/.ssh
```

Each user should run their own dirblock instance. Do not run one root-owned service expecting `~` to protect every user's home; without `SUDO_USER`, `~` resolves as root's home.

## Multi-User Behavior

dirblock marks the mount points that contain its watched directories. On a shared `/home` mount, multiple users' dirblock instances may receive fanotify events from the same mount.

This does not normally cause conflicts because each instance filters events to its own watched directories before checking process identity. If Alice's instance watches `/home/alice/.ssh`, it fast-allows opens under `/home/bob/.ssh` because that path is outside Alice's watched directories.

Avoid overlapping watched paths in managed configs. In particular, do not watch broad parent directories such as `/home` unless you explicitly want one user's dirblock instance to participate in other users' access decisions. If two instances watch the same path, or one watches a parent of another watched path, a deny from either relevant listener can block an open.

## Install Binary and Config

Build and install the binary into a system location:

```bash
make
sudo install -m 0755 dirblock /usr/bin/dirblock
```

Install the managed config:

```bash
sudo install -d -m 0755 /etc/dirblock
sudo install -m 0644 config/dirblock.toml /etc/dirblock/dirblock.toml
```

`config/dirblock.toml` should be the host or fleet policy you intend to deploy. Generate a fresh host-specific starting point with:

```bash
python3 generate_config.py
```

`config/dirblock_default.toml` is a broad reference catalog generated by `python3 generate_config.py --default`; do not deploy it blindly as a managed policy. Use it to review common variants, then deploy a tuned `dirblock.toml`.

Grant the capabilities required for fanotify and cross-user process inspection:

```bash
sudo setcap cap_sys_admin,cap_sys_ptrace+ep /usr/bin/dirblock
```

Verify:

```bash
getcap /usr/bin/dirblock
```

Expected output includes:

```text
/usr/bin/dirblock cap_sys_ptrace,cap_sys_admin=ep
```

## Config Search Order

When no `--config` argument is provided, dirblock searches:

1. `<binary_dir>/../config/dirblock.toml`
2. `~/.config/dirblock/dirblock.toml`
3. `/etc/dirblock/dirblock.toml`

For managed systems, this means users can override the managed `/etc` config by creating:

```text
~/.config/dirblock/dirblock.toml
```

That can be useful for local exceptions, but it is an administrative policy choice. If user overrides are not desired, launch dirblock with an explicit config path in the service:

```text
/usr/bin/dirblock --config /etc/dirblock/dirblock.toml
```

If user overrides are desired, launch dirblock without `--config` so the normal search order applies.

## CI/CD and Self-Hosted Runners

dirblock can be useful on persistent self-hosted CI/CD runners that keep long-lived file credentials under the runner user's home directory, such as:

```text
~/.ssh
~/.docker
~/.aws
~/.config/gh
```

It is less useful for ephemeral hosted runners, jobs that receive secrets only as environment variables, or jobs that run entirely inside container namespaces where host fanotify visibility and capabilities may not match the job's file access pattern.

Start CI/CD deployments in dry-run mode. Run dirblock as the same user that runs jobs, not as root, so `~` resolves to the runner user's home:

```bash
/usr/bin/dirblock --dry-run --config /etc/dirblock/dirblock.toml
```

For a systemd-managed runner user, use a user service with dry-run first:

```ini
[Service]
Type=simple
ExecStart=/usr/bin/dirblock --dry-run --config /etc/dirblock/dirblock.toml
Restart=on-failure
RestartSec=5s
```

If the runner must be protected when no interactive login session is active, enable lingering for the runner account:

```bash
sudo loginctl enable-linger gitlab-runner
```

Then start the user's service as that runner user:

```bash
sudo -iu gitlab-runner systemctl --user daemon-reload
sudo -iu gitlab-runner systemctl --user enable --now dirblock.service
```

Capture dry-run logs during representative builds:

```bash
sudo -iu gitlab-runner journalctl --user -u dirblock.service
```

Review `DRY-RUN DENY` lines before switching to enforce mode. CI jobs often spawn tools from runner services, shell executors, build wrappers, or container entrypoints rather than from an interactive terminal. If ancestry-profiled rules are needed, create a CI-specific profile such as `ci_runner` instead of reusing `dirblock` or `remote_ssh` broadly.

Do not enable enforce mode blindly across arbitrary jobs. First confirm:
- legitimate credential readers are allowlisted;
- build tools do not rely on broad unprofiled access to protected directories;
- containerized jobs are tested in the same mode used in production;
- DENY and DRY-RUN DENY logs are retained somewhere operators can inspect.

## User Config and Operation

When the managed service uses the normal config search order, a user can create a personal config at:

```text
~/.config/dirblock/dirblock.toml
```

That user config replaces the managed `/etc/dirblock/dirblock.toml` for that user's dirblock process. It does not affect other users. This is useful when a user needs a local allowlist exception, but administrators should decide whether user overrides are allowed for the managed environment.

After creating or editing `~/.config/dirblock/dirblock.toml`, the user must restart their own dirblock service:

```bash
systemctl --user restart dirblock.service
```

The user can stop and start their own instance:

```bash
systemctl --user stop dirblock.service
systemctl --user start dirblock.service
```

The user can check whether their instance is running:

```bash
systemctl --user status dirblock.service
```

The user can inspect DENY and DRY-RUN DENY messages in their user journal:

```bash
journalctl --user -u dirblock.service -n 100
```

If the service file changed, not just the config file, reload the user manager before restarting:

```bash
systemctl --user daemon-reload
systemctl --user restart dirblock.service
```

These commands operate on the current user's systemd user manager. They do not stop or restart other users' dirblock services.

When a user reports a legitimate denial, update only that user's local policy or the managed fleet policy, depending on where the exception belongs. [update_config.md](update_config.md) is the playbook for turning a specific `DENIED` or `DRY-RUN DENY` line into the smallest local `config/dirblock.toml` allow rule; it is not used to generate `config/dirblock_default.toml`.

## First-Login Auto-Launch

For headless managed systems, prefer a systemd user service. Do not use XDG `.desktop` autostart as the managed default; it is desktop-session oriented and is inappropriate for many SSH-only hosts.

A systemd user service starts once when the user's systemd user manager starts. Multiple SSH logins for the same user do not create duplicate dirblock processes. Logs go to the user journal, which makes DENY messages inspectable with `journalctl --user`.

Install a global user service:

```bash
sudo install -d -m 0755 /etc/xdg/systemd/user
```

Dry-run service for rollout:

```ini
# /etc/xdg/systemd/user/dirblock.service
[Unit]
Description=dirblock per-user file access policy
Documentation=man:fanotify(7)

[Service]
Type=simple
ExecStart=/usr/bin/dirblock --dry-run
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=default.target
```

Enable it globally for users:

```bash
sudo systemctl --global enable dirblock.service
```

The service starts for each user at their next login. Existing logged-in users can start it manually:

```bash
systemctl --user daemon-reload
systemctl --user start dirblock.service
```

After reviewing dry-run logs and tuning policy, change `ExecStart` to enforce mode:

```ini
ExecStart=/usr/bin/dirblock
```

Then reload user managers. For the current user:

```bash
systemctl --user daemon-reload
systemctl --user restart dirblock.service
```

For a managed fleet, apply the service file update through your configuration management and let users pick it up on next login, or restart user services according to local operational policy.

## Logging and Operations

Users can inspect service state:

```bash
systemctl --user status dirblock.service
```

Users can inspect logs:

```bash
journalctl --user -u dirblock.service
```

During dry-run rollout, look for lines like:

```text
DRY-RUN DENY: pid=... exe=... -> /home/user/.ssh/config
```

In enforce mode, look for:

```text
DENIED: pid=... exe=... -> /home/user/.ssh/config
```

Use these logs to add the smallest appropriate allow rule. Prefer exact executable paths and narrow prefixes. For runtimes such as Bun, Node, or Python, prefer cmdline filters when there is no stable native executable.

If a denial involves a tool whose executable lives inside the watched directory, test carefully in enforce mode. The shell may be the process opening the executable before `execve`, so allowing the target binary itself may not prevent self-blocking.

## Optional Lingering

By default, a systemd user service runs while the user has an active login session. For many managed SSH systems, this is the desired behavior: dirblock starts at first login and avoids duplicate processes across later SSH sessions.

If an administrator wants dirblock to run for a user even without an active login session, enable lingering:

```bash
sudo loginctl enable-linger USER
```

Disable lingering:

```bash
sudo loginctl disable-linger USER
```

Use lingering selectively. It keeps the user's systemd manager alive after logout, which may have broader resource and policy implications beyond dirblock.

## Policy Guidance

Use `~` for per-user sensitive paths in managed configs:

```toml
"~/.ssh" = [...]
"~/.gnupg" = [...]
"~/.config/gh" = [...]
```

Avoid absolute user paths in `/etc/dirblock/dirblock.toml` unless the rule is intentionally for one account only.

Avoid broad watched directories:

```toml
# Avoid on multi-user systems unless intentionally global.
"/home" = [...]
```

Do not add broad unprofiled readers, editors, shells, or runtimes to sensitive directories. Prefer:

```toml
"/usr/bin/cat;dirblock"
"/usr/bin/less;remote_ssh"
"claude:~/.bun/bin/bun"
```

Use dry-run before enforcing:

```bash
/usr/bin/dirblock --dry-run --config /etc/dirblock/dirblock.toml
```

Exercise normal workflows, review logs, tune policy, then switch the service to enforce mode.

Treat noisy or self-blocking watches as explicit opt-ins. `~/.pki`, `~/.config/git`, and `~/.cargo` can contain sensitive material, but they are not good managed defaults for every fleet: browsers and Electron apps commonly touch `~/.pki`, many tools read `~/.config/git/gitignore`, and standard Rust installs put `cargo` and `rustup` under `~/.cargo/bin`. Prefer Cargo credential providers such as `cargo:libsecret` over watching all of `~/.cargo` when the goal is protecting registry tokens.

## Troubleshooting

No config found:

```text
no config file found. Use --config or place dirblock.toml in ~/.config/dirblock/ or /etc/dirblock/
```

Install `/etc/dirblock/dirblock.toml`, create a user config, or launch with `--config`.

fanotify permission error:

```text
ERROR: fanotify_init: Operation not permitted
```

Check capabilities:

```bash
getcap /usr/bin/dirblock
```

Reapply if needed:

```bash
sudo setcap cap_sys_admin,cap_sys_ptrace+ep /usr/bin/dirblock
```

Service not starting at login:

```bash
systemctl --user status dirblock.service
systemctl --user list-unit-files | grep dirblock
```

Confirm the global service exists:

```bash
systemctl --global is-enabled dirblock.service
```

Unexpected denials:

```bash
journalctl --user -u dirblock.service --since today
```

If an ancestry-profiled rule is denied, dirblock logs an observed candidate profile. Use that output to decide whether to add a separate named profile, such as `remote_ssh`, rather than broadening an unrelated profile.
